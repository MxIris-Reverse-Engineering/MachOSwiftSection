# SwiftLayout：静态 Field Offset 计算引擎实现说明

> 配套调研见 [`StaticFieldOffsetComputation.md`](StaticFieldOffsetComputation.md)。本文记录**实际落地的实现**、与调研的差异，以及首期覆盖范围与已知降级。面向维护者。

## 背景与目标

`feature/swift-diffing` 的静态 ABI 分析需要**不加载进程、不调用 runtime**，纯从 Mach-O 文件离线算出 Swift struct/class stored-property 的准确字节偏移。`SwiftLayout` module 实现这一能力。

## 关键设计决策

1. **重算，而非读 vector（路径 B，非路径 A）。** 实测确认两点颠覆了「直接读 metadata field-offset vector」的捷径：
   - **metadata 对象的 materialize 只有一条路：runtime accessor**（`descriptor.metadataAccessorFunction(in:)` → `accessor(request:)` **执行函数**），只能在已加载 MachOImage 下做。纯 MachOFile 不能执行 accessor，无法定位完整 metadata。
   - **ValueWitnessTable 在纯 MachOFile 静态上下文不可读**（runtime 对象，指针指向运行时分配内存）。

   因此引擎离线移植 runtime 的 `performBasicLayout`：从 `FieldDescriptor` 逐字段求 `(size, alignment)` 再累加。这套求解器统一处理 fixed + resilient + 嵌套，并能正确处理编译器写 0 占位的情况。

2. **独立 `SwiftLayout` module**，依赖 `MachOSwiftSection` + `SwiftInspection`，是 `SwiftInspection` 之上的独立 peer。不依赖 `SwiftDeclaration`，使 `SwiftDiffing` 后续可按需依赖而无环。

3. **逐字段降级。** 单个字段类型解析失败（existential、actor 默认存储、跨模块 resilient、未替换泛型参数）时，该字段及其后字段标 `FieldResolution.unknown`，而非整型失败；前序已算字段仍准确返回。`AggregateFieldLayout.computedFieldOffsets` 给出可信前缀，供 diffing 只比对已算字段。

## 核心算法

- **`BasicLayout.compute`**：照搬 `Metadata.cpp:2321-2360`。逐字段 `offset = roundUpToAlignMask(accumulator, fieldAlignMask)`，累加用 **size 不是 stride**，尾部 padding 只进 stride。struct 起点 0；class 起点 = 父类 instance size（= 父类 `size`，根类 16 = `HeapObject`），递归父类；tuple 起点 0。**同时求整体 XI = 字段中最大者**（`swift_initStructMetadata`：“use the field with the most”，与 tuple 同规则），随 `AggregateLayout` 带出；`asStaticTypeLayout()` 直接采用。**曾漏此步致 struct XI 恒 0**：`asStaticTypeLayout(extraInhabitantCount:)` 默认 0、`computeStructLayout` 从不传播，于是**任何 struct 的整体 XI 都记成 0**。后果是「payload 为带 XI 的 struct（如包类引用的 `SwiftUI.Font`）」的**单 payload 枚举**误加一个 tag 字节（`Text.Style.TextStyleFont` 本应 8 字节算成 9/stride 16），并把外层聚合的后续字段偏移整体推移。修复即把 tuple 早已做对的 max-over-fields 下放进 `compute`，struct/tuple/class 共用（class 聚合的该值不作为 class *值* 的 XI，无人读取，无害）。class 值作字段时仍取 `.pointerSized`（引用 XI），不受影响。
- **`StaticTypeLayoutResolver`**：按 `Node.Kind` 分派。`builtinTypeName` → KnownLayoutTable / `builtinPrimitiveLayout`（含 `Builtin.DefaultActorStorage` 特判，见下）；`class`/`boundGenericClass` → 一个指针（**不递归字段，天然破环**）；`structure` → **BuiltinTypeLayoutIndex 回退**（C-imported 值类型，见下）或 known 表 / 递归 descriptor；`enum` → EnumLayoutBridge；`protocolList`·`protocolListWithAnyObject`·`protocolListWithClass`·`existentialMetatype`·`symbolicExtendedExistentialType`（受约束 existential） → ExistentialLayoutBridge；`dependentMemberType`（关联类型） → DependentMemberTypeBridge；`builtinFixedArray`（含 `InlineArray` 特判） → fixed-array 公式；`tuple`/`functionType`(thick=2 words)/`cFunctionPointer`·`objCBlock`·`escapingObjCBlock`(单指针=1 word)/`weak`·`unowned`·`unmanaged`(1 word)/`metatype`(thin=0/thick=8) 各自处理；其余抛 `unknown` 降级。memoize 按限定名，`inProgressKeys` 防环兜底。
- **`EnumLayoutBridge`**：no-payload（最小 tag 字节）+ single-payload（含 `Optional`，用 payload 的 extra inhabitants 编码空 case，溢出才加 tag 字节）。`getEnumTagCounts` 公式从 runtime `EnumImpl` 移植。**multi-payload enum** 两条路解析：①首选 `BuiltinTypeLayoutIndex` 整体布局（编译器 emit 的精确值，见下）；②无 builtin descriptor 时**结构化兜底**——`multiPayloadEnumLayout` 复用 `SwiftInspection.EnumLayoutCalculator`（`GenEnum.cpp`/`TypeLowering.cpp` 的离线移植）：payload 区 = 最大 payload case 的尺寸，tag 编码在公共 spare bits（取自该 enum 的 `MultiPayloadEnumDescriptor`，`__swift5_mpenum` 段）或退而附加 extra tag bytes（`calculateTaggedMultiPayload`）。indirect case 的 payload 按单指针计。**泛型实例化（环境非空）恒走 tagged 路径且跳过 builtin/spare-bits**：runtime 的 `swift_initEnumMetadataMultiPayload` 对泛型 multi-payload enum 一律附加 tag 字节（spare-bit 布局需要编译期 payload 知识），且 tag 字节的**未用值即 extra inhabitants**（`(1 << (numTagBytes×8)) − numTags`，4 字节 tag 饱和，capped `MaxNumExtraInhabitants`）——两者均已移植，`Optional<Environment<Bool>.Content>` 形态因此精确（阶段 8）。
- **`BuiltinTypeLayoutIndex`（整体布局回退）**：`__swift5_builtin` 段的 `BuiltinTypeDescriptor` 记录了「reflection 无法结构化推导布局」类型的**整体布局**（size/stride/align/XI/bitwiseTakable）——即**imported C 值类型**（`__C.CGRect`/`__C.Decimal`/…）与 **multi-payload enum**。编译器在**每个反射性引用该类型的镜像**（如把它当存储字段的类型所在镜像）里都 emit 一份，故「使用方镜像」必带。求解器在 `structureLayout`/`enumLayout` 里对**非泛型** `structure`/`enum` 节点先查 `originImage.builtinLayoutIndex`：命中（C 类型、multi-payload enum）即返回该整体布局；未命中（普通 struct/enum 不 emit builtin descriptor）落回结构化路径。对「作为字段」的场景只需整体 size/align/stride，故无需 C struct 内部偏移、也无需 multi-payload 的 spare-bit 分析。descriptor 的 `typeName` 是**符号引用**（裸串为空），故按 demangle 还原的限定名建 key（与求解器查找侧同一格式）。语义同 resilient：「针对这组具体二进制」。
- **`GenericArgumentEnvironment`（具体 bound-generic 实例化作字段，阶段 5）**：`MyBox<Int>` 作字段时，基础 generic descriptor 的字段记录存的是依赖参数（`A` = `dependentGenericParamType(depth,index)`）。做**纯语法 Node 替换**、不调任何 metadata accessor / PWT：从 `boundGeneric*` 节点自带的 `typeList` 按**位置**取具体实参（depth-0 下 index i ↔ typeList[i]），建 `(depth,index)→Node` 映射；解析字段/payload/超类类型前，先 demangle 再用一个 `Node.Rewriter`（Demangling）把节点里的 `dependentGenericParamType` 深度替换为具体实参节点，然后照常 `layout(forTypeNode:)`。深度替换意味着 `(A,B)`、`Pair<A,A>` 等也一并替换，递归再自然重建子环境；`structureLayout`/`enumLayout` 命中 `boundGeneric*` 时按节点建环境、并以**实例化 key**（remangle）memoize，使 `Foo<Int>`/`Foo<String>` 区分缓存与防环（`memoizedInstantiationLayout` 跳过 KnownLayoutTable 探测）。**仅限 depth-0 参数**；实参可为纯类型、**value（`.integer`/`.negativeInteger`，SE-0452）**或**扁平 pack（SE-0393）**——仅当 pack 实参自身仍含未展开 expansion（未解析的转发形态）时环境才整体降级为空（避免位置错配），depth>0 的参数保持降级。`class Sub<T>: Base<T>` 经 `superclassStartLayout` 先把超类节点按子类环境替换、再据替换后的 `Base<Int>` 建超类环境递归。**纯语法的副带修复**：单 payload 泛型枚举 `enum E<First, Second>{case a(Second)}` 现按 payload 字段记录 + 环境替换取**正确参数**（旧逻辑无条件取 `typeList.first` 会取错第 0 个实参）。
- **Pack expansion 展开（阶段 6）**：替换递归改为**自定义 top-down**（弃用 bottom-up 的 `Node.Rewriter`）——展开是**上下文相关**的：expansion 实例 i 里的 pack 参数引用要解析成 pack 的第 i 个元素而非整包；bottom-up 会先把 pattern 里的引用替换成整个 `.pack` 节点，与 `(repeat Pair<each T>)` 这类 pattern 中的**字面 pack** 无法区分。`.tuple` 节点：packExpansion 元素按 count（替换后须为具体 `.pack`）逐元素实例化 pattern（经 `packElementEnvironment(atIndex:)` 把 pack 绑定缩窄为第 i 元素）后拼接；空 pack → 空 tuple；展开后**恰好 1 个无标签元素 → 坍缩为该元素本身**（一元素无标签 tuple 不存在，runtime 对其直接返回元素 metadata）。`.pack` 字面量：含 expansion 的子节点（`Foo<repeat each T>` 转发形态的 `Pack{PackExpansion(…)}`）展开拼接成扁平 pack。count 未具体绑定的 expansion 原样保留，交给求解器按 `genericParameterUnsubstituted` 降级。类型侧的简化前提（编译器强制）：**一个泛型类型最多声明一个 type pack、enum 不能声明 type pack**。
- **Fixed array / value generics（阶段 6）**：求解器新增 `builtinFixedArray` 分派（`Builtin.FixedArray<count, Element>`）：`count ≤ 0` → empty；否则 **size = stride = element.stride × count**（count = 1 也不回收尾部 padding）、align/bitwiseTakable 随元素、**XI 取首元素**——从 runtime `swift_getFixedArrayTypeMetadata`/`FixedArrayCacheEntry::tryInitialize` 移植，与 IRGen `convertBuiltinFixedArrayType`、RemoteInspection `ArrayTypeInfo` 三方一致。`Swift.InlineArray<count, Element>` 与其唯一 stored field `Builtin.FixedArray` 布局恒等，在 `structureLayout` 里特判直接走同一公式（descriptor 在 stdlib，与 `Optional` 特判同理，单镜像可算；依赖闭包可达 stdlib 时结构化路径 + 环境替换算得同一结果）。tuple 的 XI 同步修正为**取元素中最多者**（runtime `swift_getTupleTypeMetadata` 语义，旧实现恒 0）——`Optional<(Int16, Bool)>` 不再多算 1 个 tag 字节。
- **`DependentMemberTypeBridge`（关联类型，阶段 7）**：泛型类型的字段用关联类型（`struct AssociatedTypeStorage<C: Collection> { var index: C.Index }`，字段串 `A.Swift.Collection.Index`）时，实例化替换只能把 base 的 `dependentGenericParamType` 换成具体类型，节点仍是 `dependentMemberType(具体base, Protocol.Assoc)`。此桥从节点取出 (base 限定名, 协议限定名, 关联类型名) 三元组，查 `__swift5_assocty` 的 witness——`ImageReference` 新建「`conforming|protocol|assoc` → `AssociatedTypeRecord`」索引，`ImageUniverse` 新增第四个解析 seam `resolveAssociatedTypeWitness`（与前三个共用惰性折入）。取到 witness 的 `substitutedTypeName` 后**在声明该 conformance 的镜像里** demangle（其符号引用相对该镜像），再用 base 自身的 bound-generic 环境二次替换（`Array` 的 `Element` witness 是 `Array` 的参数 (0,0) → `Int16`），最后照常 `layout(forTypeNode:)`。base 仍未替换（裸 `A` / depth>0）则降级。
- **受约束 / 扩展 existential（阶段 7）**：`any Boxed<Int>`（带 primary associated type 约束或 same-type 约束的 existential）经 `symbolicExtendedExistentialType` 节点编码——`symbolicExtendedExistentialType → (unique|nonUnique)…ShapeSymbolicReference → constrainedExistential → Type → ProtocolList`。**约束不改变容器大小**（`any Boxed<Int>` 与 `any Boxed` 布局相同：opaque buffer + metadata + 每协议 1 witness），故 `extendedExistentialLayout` 取出内层 `ConstrainedExistential` 的 existential 类型节点，路由回既有 `existentialLayout`（自动处理 class-bound / opaque 区分），requirement list 忽略。
- **嵌套于特化泛型父类的类型命名（阶段 7）**：`Outer<Int>.Inner` 作字段时，字段节点是 `Enum/Structure(boundGenericStructure(Outer<Int>), Identifier('Inner'))`——父上下文是 `boundGeneric*`。原 `NodeTypeNaming.contextQualifiedName` 的 switch 未处理 `boundGeneric*`（落到 `default: node.text` 返回 nil），导致父链被丢、限定名退化成裸 `Inner` 而 miss 索引（实测 SwiftUI `@Environment<Bool>.Content` 等 279 例）。修复：`contextQualifiedName` 对 `boundGeneric*` 解包到底层 nominal 再递归、对 `.type` 包裹解包。
- **父链实参环境（阶段 8）**：阶段 7 修好了嵌套类型的**命名**，但字段**使用父参数**的嵌套类型（`Environment<Value>` 唯一字段 `content: Content`，`Content` 的 payload 引用 `Value`）仍降级——`Content` 自身不带 `typeList`，实参挂在父链的 `boundGeneric*` 节点上，旧 `make(forBoundGenericNode:)` 只认节点顶层的 `typeList`，建出空环境。修复：`make(forInstantiatedTypeNode:)` 沿 nominal 父链**逐层收集**各 `boundGeneric*` 的 direct `typeList`（demangler 恰好只给声明参数的层套 `boundGeneric*` 壳），最外层 = depth 0——`Environment<Bool>.Content` 绑定 `{(0,0)→Bool}`，多级 `Outer<Int8>.Inner<Int64>` 绑定 `{(0,0)→Int8, (1,0)→Int64}`（顺带修正了旧实现对多级实例化把最内层 typeList 误绑 depth 0 的潜在错位，此前被降级掩盖）。`structureLayout`/`enumLayout` 的 builtin 查表改为**在环境为空时**才做（此前按 `node.kind` 判非泛型；`Enum(BoundGeneric…, Content)` 的 kind 是裸 `.enum`，需以环境判定实例化）。实测这是系统框架剩余 198 例 `genericParameterUnsubstituted` 的全部根因（SwiftUI 的 `Environment<…>` 字段族）。
- **class-bound 泛型参数不特化解析（阶段 9）**：阶段 5–8 都在解**具体实例化**（`Foo<Int>` 有实参）。阶段 9 转向**未特化**的泛型类型本身：一个 `T: AnyObject` / `T: SomeClass` / `T: SomeClassBoundProtocol` 约束的参数，其存储在运行期**必然是单个对象引用**（8 字节、8 对齐、bitwise-takable、XI = `swift_getHeapObjectExtraInhabitantCount` = 0x1000，与具体 class field 完全一致），且**每个 class 实例化布局都相同**——故 `T` 字段本身及其后所有字段无需任何实参即可精确算出。`ClassBoundGenericParameterAnalysis` 读 descriptor 的**累积规范需求签名**（`genericContext.requirements`；需求的 subject 携带绝对 `(depth,index)`，与字段记录同坐标系，嵌套上下文无需额外簿记），demangle 每条需求的**裸参数** subject（约束在依赖成员 `T.Element` 上的不 pin `T`，跳过），再按需求 kind + resolved content 判 class-bound：`layout(.class)`（`T: AnyObject`）、`baseClass`（`T: SomeClass`）、或 class-bound `protocol`（`.objc` 恒真；`.swift` 看 descriptor 的 `classConstraint`；跨镜像 symbol 引用经 `resolveProtocolClassConstraint` + `isObjCProtocolDeclared` seam 按名回收）。结果经 `augmentedWithClassBoundParameterKeys` 注入 `GenericArgumentEnvironment.classBoundParameterKeys`，在每个读字段入口应用（`StaticLayoutCalculator.fieldLayout(ofStruct:/ofClass:)` 与求解器 `computeStructLayout` / `computeClassLayout` 收口点——故泛型超类 `class Sub<Element: AnyObject>: Base<Element>` 与 ObjC 祖先泛型子类同样受益）。替换阶段：未被具体实参替换、且命中 class-bound 集的 `dependentGenericParamType` 经**同一套 top-down 替换**（故能钻进 `Element?`/`(Element, Int)`/`[Element]`/闭包等复合类型）改写成占位 `.class` 节点，求解器按既有 `.class → .pointerSized` 分派（占位节点名字惰性——class 节点从不被读名/递归）。**具体实参始终优先**于 class-bound 兜底（class-bound 参数的具体实例化用其真实实参）；class-bound 泛型 **multi-payload enum** 仍走 tagged 分支（注入后环境非空，spare-bits 路径按实例化规则关闭）。剩余不可静态定：无约束或非 class-bound 参数（`T`、`T: Equatable`、value/pack 参数）——不特化即缺实参，保持逐字段降级。
- **具体 sameType 钉住（阶段 9 补全）**：同一趟需求扫描里，除 class-bound 外还提取 `sameType` 钉住。`extension Foo where Value == Date { struct Bar { var x: Value } }` 这类**约束扩展**里嵌套的类型，其 descriptor 的泛型签名继承了外层 `Value` 参数**加上** `Value == Date` 需求，故字段 `x: Value` 不特化即解析为 `Date`。`ClassBoundGenericParameterAnalysis.layoutFacts` 现返回 `RequirementSignatureLayoutFacts { classBoundParameterKeys, concreteSameTypeSubstitutions }`：对 kind==`.sameType` 且 LHS 为**裸参数**的需求，取 RHS（`resolvedContent` 的 `.type(MangledName)`）demangle、**仅当其完全具体**（不含 `dependentGenericParamType`/`dependentMemberType`——引用别的参数则无法独立替换）时，把 unwrapped RHS 节点作为**真正的 substitution**（不是占位 class）注入 `GenericArgumentEnvironment.substitutions`（`augmented(withRequirementFacts:)`，仅填实例化节点未覆盖的参数——真实例化实参恒优先，且与钉住不矛盾）。钉住到具体类型的参数被编译器从 accessor 的 key 实参里**丢弃**（已被完全确定），故这类嵌套类型的 runtime 布局用**无参** accessor 取得。RHS 可为标量（`Int64`，单镜像即解析）或 bound-generic 具体类型（`Range<Int>`，经依赖闭包解析 stdlib descriptor）。实测**收益很小**（5 框架 579 条 sameType 里仅约 23 条是「裸参数 == 具体类型」——75% 是依赖成员 subject 归 assocty bridge、21% 是抽象参数对参数），但它是需求签名能不特化推断的**正确补全**：class-bound + 具体 sameType 合起来 = 需求签名不带实参所能确定的全部。**同值钉住（value generic）同样打通**：`extension ValueGenericBuffer where count == 5` 的需求即 kind==`sameType` 加 `isValueRequirement` 标志（GenMeta.cpp:7460——编译器复用 sameType 而非新增 SameValue kind），RHS mangle 为整数字面量（demangle 出 `.integer` 节点）——`.integer` 天然通过「完全具体」检查、环境按阶段 6 的 value 实参规则绑定，**零代码改动即解析**（fixture `ValuePinnedInner` 的 `InlineArray<count, Int16>` 字段 10 字节，双 reader 对 runtime 逐字段相等；值钉住参数同样被丢出 accessor key 实参，无参 accessor 即真值）。顺带修复 SwiftPrinting 接口打印**丢 SE-0452 整数节点**的缺陷（`NodePrintable.printNameInBase` 补 `.integer`/`.negativeInteger`——此前 `ValueGenericBuffer<>`、`InlineArray<, Swift.Bool>`、`where A == ` 渲染为空且已固化进旧接口快照；SwiftDump 路径走 Demangling `NodePrinter` 一直正确）。
- **Swift 声明 `@objc` 协议的 existential（阶段 8）**：`PlatformAccessibilityElementProtocol & NSObject` 形态。Swift 声明的 `@objc` 协议**不 emit Swift protocol descriptor**（不进 `__swift5_protos`），唯一 runtime 痕迹是 ObjC protocol record（`__objc_protolist`，旧式 `_TtP<module><name>_` 名）。`ExistentialLayoutBridge` 在 class 约束查找 miss 后回退查第五个 seam `isObjCProtocolDeclared`（`ObjCProtocolIndex` 把 `_TtP…_` 解析回限定名建索引；原生 ObjC 协议的裸名不解析——它们 demangle 成 `__C` 引用、在更早的分支处理）：命中即视同 imported ObjC 协议——强制 class-bound、不计 witness table。实测这是系统框架剩余 13 例 `typeDescriptorNotFound` 的全部根因。
- **`ExistentialLayoutBridge`**：从 runtime `ExistentialTypeInfoBuilder`（`TypeLowering.cpp`）移植。opaque `any P` / 协议组合 = 3-word inline buffer + 1 metadata word + 每协议 1 witness word（`32 + 8N`）；class-bound（`AnyObject` / class 约束协议 / 显式 superclass）= 1 object word + N witness（`8·(1+N)`）；`any Error` = 1 boxed word（8）；existential metatype = 1 metadata word + N witness（`8·(1+N)`，与 class-bound 无关）。是否 class-bound 由各协议 descriptor 的 class 约束决定（`protocolListWithAnyObject`/`WithClass` 结构上即 class-bound）。**marker 协议（`Sendable` 等）已被编译器从 mangled field 名剥离**，故每个列出的协议都计 1 个 witness，无需自行过滤 marker。跨模块协议的 class 约束现经依赖闭包解析（见下）。**imported ObjC 协议**（`any NSCopying` 等，由 `NodeTypeNaming.objCProtocolBareName` 识别 `__C.<Name>` 的 `.protocol` 节点）无 Swift descriptor：它**恒为 class-bound 且不贡献 Swift witness table**（`id<P>` 即单个 class 引用），故强制 existential class-bound 且不计入 witness 数——纯 ObjC existential = 8 字节，混合组合 = 1 object word + N(Swift 协议) witness。
- **ObjC 祖先起点（阶段 4）**：`superclassStartLayout` 中，超类 demangle 出的若是 `__C.<Name>`（ObjC 类，由 `NodeTypeNaming.objCClassBareName` 识别），则该类自身字段从 ObjC 父类的 `instanceSize` 起算（`NSObject` = 8，仅 isa），经第三个 seam `resolveObjCClassInstanceSize(byBareName:)` 取得；**先于** Swift 侧 `resolveType` 判断（ObjC 名在 Swift 类型索引里必然 miss，若先走 `resolveType` 会无谓地把整条依赖闭包索引一遍）。读法用 `ObjCClassIndex`：从 `__objc_classlist` 取 instance `class_ro_t`（in-process 的 realized 类经 `class_rw_t` 回退）的 `instanceSize`——与 `ObjCClass.info(in:).instanceSize` 同值但不解析 methods/ivars；**取 `instanceSize` 而非 `instanceStart`**（后者是 ObjC 类自身首 ivar 起点，cache 上常为 0）。alignmentMask 取 7（指针对齐）。ObjC 类无 Swift descriptor，故只能这样起算；其上的 Swift 字段照常累加。
- **`ImageUniverse` 依赖闭包（阶段 3）**：`singleImage` 之外新增 `dependencyClosure`。求解器始终只经 `resolveType` / `resolveProtocolClassConstraint` / `resolveObjCClassInstanceSize` 三个 seam 查类型，故闭包**不动求解器一行**。root 立即索引，依赖按 BFS 顺序**惰性索引**——仅当某次 resolve 在已索引镜像里全部 miss 时，才推进索引下一个依赖、命中即停（一次 OS 全闭包可达数百镜像，eager demangle 每个会耗数秒；真实查询只命中前几个 Swift 依赖）。便利工厂：`MachOImage` 经活动 dyld（`MachOImage(name: bareName)`）解析，`MachOFile` 经显式 on-disk 路径 + dyld shared cache（cache 一次性按 bare-name 建索引）。详见 [StaticLayoutDependencyClosure.md](StaticLayoutDependencyClosure.md)。

## 验证

`StaticLayoutVsRuntimeTests` 遍历 fixture 二进制（`SymbolTestsCore`）里**每个非泛型 struct/class**，以 runtime metadata accessor 读出的 field-offset vector 为 ground truth，断言静态引擎重算**逐字段相等**（引擎自身从不调用 accessor）。降级类型则断言**已算前缀**与 runtime 前缀一致。`ExistentialLayoutTests` 另以**字面值**锁定 existential / actor 类型的完整 field-offset 向量（如 `ExistentialFieldTest = [0, 40, 80, 120, 128, 136]`），独立于宽泛前缀套件，精确钉住容器尺寸公式。

单镜像当前结果：**147 个类型参与比较，142 个完全算对，0 个 mismatch**；其余 5 个为单镜像下的合理降级（ObjC 祖先 2 个 + 跨模块 resilient/字段 3 个）——其中 ObjC 祖先 2 个与跨模块 3 个均由依赖闭包（阶段 3/4）解析掉，见下。覆盖率下限断言（`comparedCount > 100`、`fullyComputedCount > 140`）防止「全降级静默通过」并锁定 existential + actor + 具体 bound-generic 实例化收敛带来的提升。具体 bound-generic 实例化作字段另由 `GenericInstantiationLayoutTests` 专门验证：9 个非泛型包裹类型（字段含 `GenericStructNonRequirement<Int>`/`<String>`、`Pair<Box<Int>, Int>`、`Box<Int>?`、元组、单/多 payload 泛型枚举、具体泛型超类的子类等）全部**完全解析**且逐字段等于 runtime offset；`GenericArgumentEnvironmentTests` 以手搓 Node 树单测替换与 value 实参降级守卫。**顶层**具体实例化另由 `TopLevelGenericInstantiationLayoutTests` 验证：`fieldLayout(of:genericArguments:)` 对 `GenericStructNonRequirement<Int>`/`<String>` 与 `GenericClassNonRequirement<Int>`、`fieldLayout(forInstantiationMangledName:)` 对一条**二进制里真实的 `GenericStructNonRequirement<Int>` 字段引用**，均**完全解析**且逐字段等于 runtime **特化** metadata 向量（经 accessor 传具体实参 metatype 取真值），并守卫 value 实参降级（裸 `field2: A` 后续字段转 `unknown`，前序 `field1: Double` 仍算对）。

## 实测发现（与调研的差异）

- **建「限定名 → descriptor」索引必须用 `MetadataReader.demangleContext(for: ContextDescriptorWrapper)`**，而非 `demangleType(descriptor.mangledName)`——后者对 descriptor 自身的 mangledName 解析失败（它不是可 demangle 的类型引用）。
- **`EnumLayoutCalculator.LayoutResult` 不含 enum 整体 `(size, align, stride)`**，故 enum 整体 size/stride 由 `EnumLayoutBridge` 自行移植 runtime 公式得出，未复用 `calculateSinglePayload` 的返回值。**XI 则相反**：三条策略的 `extraInhabitantCount` 现由 calculator 在各自 `calculate*` 内按参考实现计算并随 `LayoutResult` 携带（spare-bits/混合按 IRGen `MultiPayloadEnumImplStrategy::getFixedExtraInhabitantCount`、tagged 按 runtime `swift_initEnumMetadataMultiPayload`、single-payload 取 payload 剩余），bridge 直接采用——GenEnum 移植知识集中在 port 一处。
- **metatype 区分 thin/thick（阶段 9+）**：一个 metatype 字段是 thin（0 字节，无存储）还是 thick（8 字节，单个 metadata 指针，XI=0x1000）由**字段记录里 metatype 实例节点的语法种类**决定，而**非**聚合体是否泛型。规则：实例是**具体 value 类型**（struct/enum/tuple/builtin）→ thin（IRGen 直接引用其 metadata）；实例是 **class 或泛型参数/依赖成员**（archetype）→ thick。archetype 情形在**每个实例化里都固定为 8 字节**（实测 `Element.Type` 在 `Foo<Int>` 与 `Foo<Int8>` 中同为 8——IRGen 在 metadata pattern 期以 archetype 一次性 lower 该字段），故泛型类型里 metatype-of-参数 的字段**不特化即精确解析**，而同一泛型类型里字面的具体 metatype（`Int64.Type`）仍为 thin。为此 `GenericArgumentEnvironment.substitute` **刻意不替换** `.metatype` 节点内部（否则 archetype 的 `T.Type` 被替成 `Int.Type` 会误判为 thin）。曾有一版错误模型以「聚合体是否 runtime-instantiated」判定 thinness，被 `GenericMetatypeFieldStruct` fixture 实测推翻（泛型类型里的具体 `Int64.Type` 仍 thin），改回按实例种类判定后两种 lowering 世界均与 runtime 吻合，且清零普查里残余的 `metatype(DependentGenericParamType)` 降级。
- **`Optional<T>` 特判**：其 descriptor 在 stdlib 不在本 image，直接从 payload 类型按 single-payload（1 空 case）计算。
- **marker 协议在 mangled field 名里已被剥离**：`any (P & Sendable)` 的 field 类型 demangle 后只剩 `P`（`sendableComposition` 实测），故 witness 计数直接数协议个数即可，无需识别/排除 marker。
- **existential class-bound 不能只看 Node 形态**：`any ClassBoundProtocolTest` 的 Node 是普通 `ProtocolList`（非 `WithAnyObject`），但结果是 16 字节 class existential。必须解析协议 descriptor 的 class 约束（`ProtocolContextDescriptorFlags.classConstraint`）。为此 `ImageReference` 额外建「协议限定名 → class 约束」索引；协议在 `__swift5_protos`（经 `protocolDescriptors`），与类型的 `__swift5_types`（`contextDescriptors`）是**不同 section**，须分别遍历。
- **`Builtin.DefaultActorStorage` = 96 字节、对齐 16**：`NumWords_DefaultActor`(12) × pointer size，对齐为 pointer 对齐的 2 倍。本 fixture 未 emit 其 builtin descriptor，故在 `builtinPrimitiveLayout` 按常量给出，移植自 runtime reflection lowering 的 `getDefaultActorStorageTypeInfo()`。actor 类型（如 `ActorTest = [16, 112]`）由此完全解析。
- **`TypeLowering.cpp`（RemoteInspection）是官方离线 lowering，是本引擎的权威对照**：existential 与 default-actor 公式均以它为准，并经 runtime accessor 真值交叉验证。
- **`MachOFile.imagePath` 返回的是 install name（LC_ID_DYLIB），不是磁盘路径**：框架的 install name 是 `@rpath/Foo.framework/.../Foo`。故 `MachOFile` 闭包的显式 search path 不能由已加载 root 的 `imagePath` 字符串替换得出，须另给真实磁盘路径；依赖名匹配统一归一到 **bare name**（末段去扩展名，即 `MachOImage(name:)` 的语义）。
- **闭包须容忍缺段**：纯 ObjC/C dylib（libobjc、libsystem_*）无 `__swift5_types`/`__swift5_protos`，许多镜像无 `__swift5_builtin`。`ImageReference`/`BuiltinTypeLayoutIndex` 把 `sectionNotFound` 视作「该镜像无此类内容」返回空，而非让整个闭包构建失败。
- **依赖闭包须惰性索引**：`SymbolTestsCore` 的传递闭包实测达 **551 个镜像**；若 eager 索引（逐镜像 demangle 全部 descriptor）约 **13 秒**。改为 root 立即索引、依赖按 BFS 顺序仅在 resolve miss 时逐个增量索引、命中即停后，闭包构建（收集 551 镜像）约 0.8 秒，整套闭包测试数秒内完成。一次「真 miss」（名字哪都没有）会触发把整列表索引一遍，之后全 O(1)。
- **resilient 子类无 field-offset global**：`ResilientChild`/`ResilientObjCStubChild` 既无 runtime field-offset vector（`fieldOffsets(in:)` 返回 `[]`），也**不 emit `…Wvd` field-offset global**（偏移完全运行时计算）。故其验证只能走「从跨模块父类静态 instanceSize 推导的字面值」，且语义是「针对闭包里这组具体二进制」。
- **ObjC 起点取 `instanceSize` 而非 `instanceStart`，且 realized 类不可读 `classROData`（阶段 4 的关键坑）**：调研单点 finding 主推「读 `class_ro_t.instanceStart`」，实测**否决**——(1) `instanceStart` 是 ObjC 类自身首 ivar 起点（= 其父类大小），dyld cache 上常写 0；Swift 子类字段起点应是父类 **`instanceSize`**（`NSObject` = 8）。(2) **in-process 的 `NSObject` 是 realized 类**，`classROData(in: MachOImage)` 返回 `nil`（`data` 指向 `class_rw_t`），须经 `classRWData → classROData`（或 `ext.classROData`）回退取 instance ro；offline 的 cache `MachOFile` 则 `classROData` 直接可读。唯一在 in-process 与 cache 两条 reader 上都正确且一致的值是 instance `class_ro_t.instanceSize`（= `ObjCClass.info(in:).instanceSize`，但 `ObjCClassIndex` 直接读 ro、不解析 methods/ivars/metaclass，故更轻）。
- **ObjC 索引随 Swift 索引同步惰性折入，无额外扫描**：`resolveObjCClassInstanceSize` 复用 `resolveType` 的 `indexNextDependency` 折入机制（ObjC 索引在 `ImageReference.init` 里一并构建、`mergeIndexes` 一并合并）。又因 ObjC 父类在 `superclassStartLayout` 里**先于** Swift `resolveType` 判断，ObjC 名不会触发「Swift 全 miss → 折满 551 镜像」；折到 libobjc 命中 `NSObject` 即停，比旧路径（ObjC 名走 `resolveType` 折满全闭包再抛错）更快。
- **`objc.classes64` / `info(in:)` 是 `MachOFile`/`MachOImage` 具体重载（非协议泛型）**：故 `ObjCClassIndex` 与 `ImageReference` 的 ObjC 索引构建须按 `as? MachOImage` / `as? MachOFile` 向下转型分派，复刻 `ImageUniverse+DependencyClosure` 的 `where MachO == …` 拆分写法。
- **`BuiltinTypeDescriptor.typeName` 是符号引用，裸串为空——必须 demangle 才有 key**：`__swift5_builtin` 段实测**不含** `Builtin.*` 条目，全是 imported C 值类型（`__C.Decimal`/`__C.CGRect`/…）与 multi-payload enum；其 `typeName` 指向 context descriptor（相对指针），`typeString` 为空字符串。旧 `BuiltinTypeLayoutIndex` 按 `typeString` 建 key，把全部条目塞进 key `""`（互相覆盖）→ 索引等于失效、且只在 `builtinTypeName` 分支被查（而该段无 `Builtin.*`），实为死代码。修复：用 `MetadataReader.demangleType` 还原成限定名（`__C.Decimal` / `SymbolTestsCore.Enums.MultiPayloadEnumTests`）建 key。
- **builtin descriptor 由「使用方」镜像 emit，故查 `originImage`**：编译器在每个**反射性引用**该类型的镜像里都 emit 一份 builtin descriptor（实测 fixture 自己就带 `__C.Decimal`、Foundation 也带一份）。所以解析某字段类型时查**字段所属镜像**（`originImage`，求解器已线程化）的 builtin 索引正中靶心，无需跨镜像合并。
- **builtin 回退限非实例化节点**：builtin key 是 generic-argument-free 的限定名（`nominalQualifiedName` 会剥泛型实参），查表前须确认节点未被实例化，避免 `Foo<Int>` 与 `Foo<String>` 撞到同一 key；`Optional<T>` 等泛型枚举继续走结构化/payload 路径。判据在阶段 8 从 `node.kind`（非 `boundGeneric*`）改为**环境为空**——`Enum(BoundGeneric…, Content)` 形态的 kind 是裸 `.enum`，但实参在父链上、层层可见于环境。
- **泛型替换：`MetadataReader.demangleType` 返回的是 `.type`-包裹节点，但求解器分派会先解包**：求解器 `layout(forTypeNode:)` 先 `unwrappedType` 再分派，故 `structureLayout` 收到的是裸 `boundGenericStructure`；而 `superclassStartLayout` 拿到的超类节点是**刚 demangle 的 `.type`-包裹**节点。`GenericArgumentEnvironment.make` 若不容忍 `.type` 包裹，对超类节点会 `kind == .type` → 误判非泛型 → 返回空环境 → 超类字段 `A` 不替换而降级（具体泛型超类子类用例实测踩到）。修复：`make` 先解一层 `.type` 再判 `boundGeneric*`，使「已解包」与「刚 demangle」两侧建出同一环境。
- **泛型实例化的 memoize/防环 key 必须含实参，且要绕开冻结表**：原 `memoizedNominalLayout` 按裸限定名做 key 且首步查 `KnownLayoutTable`——对 `Foo<Int>`/`Foo<String>` 会撞同 key（裸名相同）。故 `boundGeneric*` 改用 `memoizedInstantiationLayout`（key = remangle 后的实例化名，**跳过**冻结表探测）。同时 `structureLayout`/`enumLayout` 仍在建环境前**先按裸名查冻结表**：`Array<Int>`/`UnsafePointer<Double>` 等 stdlib 泛型布局与实参无关、靠裸名命中冻结表（指针大小）；若直接走实例化 key 会漏表并尝试结构化展开 stdlib 内部存储而失败（单镜像下其 descriptor 也不可达）。`Range<Int>` 不在冻结表，单镜像下降级、依赖闭包下方可结构化解析。
- **类型侧 pack 的两条编译器硬限制大幅简化实现**：`struct TwoPacks<each T, each U>` 报 "generic type cannot declare more than one type pack"，`enum E<each T>` 报 "enums cannot declare a type pack"（均实测）。故类型场景下 expansion 的 count 必然引用唯一的那个 pack 参数（多 pack 对齐问题不存在），enum 与 pack 的交互全部经由 struct/tuple payload 间接发生。
- **`packExpansion` 与 `pack` 的 wrapper 约定不同**：demangler 对 `packExpansion` 的 `[pattern, count]` 子节点是**裸的**（`popTypeAndGetChild` 已解包），而 `pack` 字面量的子节点是 **`.type` 包裹**的（`popPack`）；`integer` 在 typeList 里也是 `.type` 包裹。替换/展开两侧的 wrapper 处理必须逐一对上（`swift-demangle -expand` + 端口源码双向核实）。
- **展开必须 top-down，bottom-up `Node.Rewriter` 有歧义**：`Rewriter.rewrite` 先重写子节点再 visit 父节点——pattern 里的 pack 参数引用会先被替换成整包 `.pack`，与 `(repeat Pair<each T>)` pattern 中 `Pair` 的**字面 pack 实参**（`Pack{x}` → 替换后 `Pack{Pack{…}}`）无法区分。改为自定义 top-down 递归 + 逐 expansion 实例的 `packElementEnvironment(atIndex:)`（pack 绑定缩窄为第 i 元素）后天然无歧义。
- **零大小字段的 offset：编译器 emit 0，runtime 算累加位置**：`PackExpansionFieldHolder.empty: VariadicPack< >` 的 runtime field-offset vector 实测为 0 而非累加器位置 26——非泛型 struct 的向量由 IRGen 静态发射，`ElementLayout::completeEmpty` 把上报的 `ByteOffset` 置 0（布局用的 `ByteOffsetForLayout` 单独存放）；而 `performBasicLayout`（runtime 实例化路径）会上报累加位置。引擎在 `accumulateFieldLayout` 镜像编译器约定（size == 0 → 上报 0）；两种约定对无存储字段语义等价。
- **`Builtin.FixedArray<1, E>` 不回收尾部 padding**：`InlineArray<1, (Int64, Int8)>` 实测 size = 16（= stride(E)×1）而非 9（= size(E)）——runtime 缓存路径统一 `size = stride = stride(E)×count`，`count == 1 && size == stride` 的元数据复用捷径只在无尾 padding 时成立，两者一致。
- **tuple 的 XI 取「XI 最多的元素」**（`swift_getTupleTypeMetadata` 的 `extraInhabitantProvidingElement`）：旧引擎恒记 0，`Optional<(Int16, Bool)>` 会多算 1 个 tag 字节（实测 runtime size 3 / 旧引擎 4）。pack 展开让 tuple 字段更常见，一并修正。
- **struct 的 XI 同样取「字段中最多者」，此前漏了**（`swift_initStructMetadata`：“use the field with the most”，`Metadata.cpp:2938`）：tuple 已改对，但 struct 走 `computeStructLayout` → `asStaticTypeLayout()` 默认 XI 0，从不传播字段 XI，导致**所有 struct 整体 XI 恒 0**。这对 struct 自身的 size/offset 无影响（struct 的 XI 不进自己的 size），但当它作为**单 payload 枚举的 payload** 时，枚举因「payload 无空位」误加 tag 字节 → size 虚增。用户实报：`SwiftUI.Text.Style.TextStyleFont`（payload `Font`——一个包 `AnyFontBox` 类引用的 struct，运行期 XI = 引用 XI）本应 8 字节被算成 9/stride 16，并把 `Text.Style` 的 `fontModifiers` 从 0x08 推到 0x10（反汇编 `Text.Style.init` 里空数组写在 `[+0x08]` 佐证）。修复：max-over-fields 下放进 `BasicLayout.compute`。
- **`String` 冻结表 XI 从 1 改为 `MaxNumExtraInhabitants`（0x7FFFFFFF）**：`_StringObject` 第二字是 tagged discriminator，保留海量非法位型，运行期即 max。旧值 1（“够撑 `Optional<String>`”）对**两个以上空 case** 的 String-payload 单枚举不够——`enum{case value(String);case none;case error}`（fixture `SinglePayloadEnumTest`）只有 1 个空位可用、第 2 个空 case 溢出成 tag 字节 → size 17（应为 16）。这是同一类「under-report XI → 枚举虚增字节」的第三个实例，根源在冻结表而非 `computeStructLayout`。
- **叶子类型的 XI 值仍是近似，不追求与 runtime 精确相等**：unsafe/autoreleasing 指针运行期 XI = 1（仅 null 非法）我们记 4096（`.pointerSized`）、引用/容器类型运行期常报 max 我们记 4096 等。这些差异只影响 dump 里的 `extraInhabitantCount` 数值，**不影响 size/stride/alignment**（除非跨过某单 payload 枚举的空-case 阈值——现有 fixture 无此形态）。故 `WholeTypeLayoutVsRuntimeTests` 只精确断言 size/stride/alignment；精确叶子 XI 建模（区分 unsafe-pointer=1 / heap-ref / String=max）列为后续。多 payload 枚举 XI 已精确（`MultiPayloadEnumStructuralTests`）。
- **单元素坍缩有 runtime 真值背书**：`Pair<Int16>`（`(repeat each T)` 实例化到单元素 pack）实测 size/stride/align = 2/2/2 ≡ `Int16`，且 `Optional<Pair<Int16>>` = 3/4——一元素无标签 tuple 不存在，替换在类型层面即坍缩。
- **ABI 稳定的类型迁移不改 mangled 模块名**：`SwiftUI.Environment` 的 descriptor（macOS 15 起）物理上在 **SwiftUICore** 二进制里，mangling 与限定名仍是 `SwiftUI.Environment`——按限定名查依赖闭包天然命中（SwiftUICore 是 SwiftUI 的依赖），但**在 SwiftUI 二进制自身的 `__swift5_types` 里找不到它**（探针实测）。诊断此类问题时不要假设「模块名 = descriptor 所在镜像」。
- **泛型 multi-payload enum 恒为 tagged，且 XI 来自未用 tag 值**：`swift_initEnumMetadataMultiPayload`（Enum.cpp）对泛型实例化不看 `__swift5_mpenum` 的 spare bits（那是编译期布局的产物），一律 `size = payloadSize + numTagBytes`、`XI = (1 << (numTagBytes×8)) − numTags`（4 字节 tag 饱和、capped）；IRGen 的非泛型 tagged 公式（`getFixedExtraInhabitantCount`：`totalTags − numTags`）与之一致，故 tagged 路径统一移植一份公式即可。若泛型实例化误走 spare-bits 路径（mpenum descriptor 按裸限定名可命中泛型 enum），size 会算错——环境非空必须跳过。
- **按名 grep fixture 符号会被 Swift mangling 的词替换骗过**：`nm SymbolTestsCore | grep ValueGenericBuffer` 命中 0 **不代表**类型没编进去——mangling 对**词**做替换（`ValueGenericBuffer` 与外层 `GenericFieldLayout` 共享 "Generic" 等词根，符号里不含完整字面串），而 `SameTypePinnedOuter`、`VariadicPack` 这类无共享词的名字才能匹配。判断类型是否在 fixture 里应跑测试或用 demangle 后比对，不要 grep 原始符号。
- **Swift 声明的 `@objc` 协议只活在 `__objc_protolist`**：实测 `PlatformAccessibilityElementProtocol` 不在 SwiftUI 的 `__swift5_protos`（`@objc` 协议不 emit Swift protocol descriptor），只以旧式 mangled 名 `_TtP7SwiftUI36PlatformAccessibilityElementProtocol_` 出现在 ObjC protocol 列表。`ObjCProtocol64.mangledName(in:)` 直接给这个串；解析回 `Module.Name` 只需处理两段 `_TtP<len>module<len>name_` 形态（文件作用域声明；嵌套/`@objc(Name)` 自定义名不匹配则跳过，保持降级）。`objc.protocols64` 与 classes 一样是 `MachOFile`/`MachOImage` 具体重载，索引构建按 reader 拆分。
- **multi-payload 结构化：`EnumLayoutCalculator.LayoutResult` 无 size/stride，须自行推导**：`calculateMultiPayload`/`calculateTaggedMultiPayload` 返回的是逐 case 投影 + `tagRegion`/`payloadRegion`/`numTags`，**没有**整体 size/stride。推导：extra tag bytes = `tagRegion` 中**起点 ≥ payloadSize** 的那段长度（spare-bits 策略下 tagRegion 落在 payload 内部 → 0；tagged/混合策略下落在 payload 之后）；`size = payloadSize + extraTagBytes`，`stride = roundUp(size, maxPayloadAlignMask)`，`align = maxPayloadAlignMask`。payloadSize/align 由遍历所有 payload case、解析各 payload 类型取 max 得出。实测对 6 个 fixture multi-payload enum（含 spare-bits 17/24、21/24、25/32 与 indirect 8/8、9/16）与 runtime VWT 逐一精确吻合。

| 反射元数据被整体裁剪的二进制 | 无 field descriptor 可读，引擎无从下手 | 本质如此 |

> **系统框架实测（5 个框架、5416 个非泛型类型、10954 字段）**：阶段 7 三项修复后总字段降级率从 **10% 降到 4%**。清零的类别：`unsupportedTypeKind` 122 → 0（关联类型 114 + 扩展存在类型 7 + `Builtin.RawUnsafeContinuation` 1）、嵌套类型丢父链导致的 `typeDescriptorNotFound` 279 → 13；连带级联 `precedingFieldUnresolved` 704 → 276。
